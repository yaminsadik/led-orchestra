# Wire Protocol

Phase 2 uses a fixed-width UDP packet to send a resolved scene from the
controller to one node or to all nodes. The packet type lives in
`shared/src/protocol.rs` as `SetScenePacket`.

## Transport

- Default bus: `udp://239.42.0.1:4242`
- Unicast example: `udp://192.168.1.42:4242`
- LAN broadcast example: `udp://192.168.1.255:4242`
- Packet size: 16 bytes
- Byte order: big endian for multi-byte integers
- Broadcast target: `target_node = 0`

The controller can already send this packet:

```bash
cd controller
cargo run -- all effect rainbow --brightness 40 --speed 128
cargo run -- all solid ff8800 --brightness 40
cargo run -- --bus udp://192.168.1.42:4242 --target-node 1 all off
cargo run -- --bus udp://192.168.1.255:4242 --target-node 1 all solid ff0000
```

## SetScenePacket Layout

| Offset | Size | Field | Meaning |
| ---: | ---: | --- | --- |
| `0` | 2 | magic | ASCII `LO` |
| `2` | 1 | version | Protocol version, currently `1` |
| `3` | 1 | kind | Packet kind, `1` = set scene |
| `4` | 4 | sequence | Controller sequence value |
| `8` | 2 | target_node | `0` for broadcast, otherwise a concrete node id |
| `10` | 1 | effect_id | Stable effect wire id |
| `11` | 1 | color.r | Red channel |
| `12` | 1 | color.g | Green channel |
| `13` | 1 | color.b | Blue channel |
| `14` | 1 | speed | Animation speed, `128` = nominal |
| `15` | 1 | brightness | Final brightness, `0..=255` |

## Effect IDs

| Wire id | Effect |
| ---: | --- |
| `0` | `off` |
| `1` | `solid` |
| `2` | `rainbow` |

Effect ids are compatibility-sensitive. Add new ids at the end and never reuse
old ids for different behavior.

## Firmware Receive Rules

The Phase 2 firmware receiver:

1. Listen on UDP port `4242`.
2. Decode incoming datagrams with `SetScenePacket::decode`.
3. Ignore packets with invalid magic, unsupported version, unsupported kind, or
   unknown effect id.
4. Ignore packets where `packet.is_for_node(NODE.node_id)` is false.
5. Apply the packet scene atomically enough that the LED render loop never sees
   half-updated parameters.
6. Keep rendering the last valid scene when no packet arrives.

For Phase 2, UDP is deliberately best-effort. It is acceptable for the operator
to send a command again. Later phases can add heartbeats, state replay, and
acknowledgements if needed.

## Example Packet

Command:

```bash
cargo run -- --bus udp://192.168.1.42:4242 --target-node 1 all solid ff8800 --brightness 40
```

Payload fields:

| Field | Value |
| --- | --- |
| magic | `LO` |
| version | `1` |
| kind | `1` |
| target_node | `1` |
| effect_id | `1` (`solid`) |
| color | `ff8800` |
| speed | `128` |
| brightness | `40` |

The sequence value is generated by the controller at send time.

## Future Packet Kinds

Likely additions:

- `NodeHello`: node advertises id, firmware version, current IP, and segment.
- `SetConfig`: controller provisions `NodeConfig`.
- `SyncClock`: controller sends a time origin or clock offset.
- `Heartbeat`: node reports liveness and current sequence.
- `OtaChunk`: possible Rust fallback update payload. The primary offline OTA
  direction is now Matter OTA in Phase 6.

New packet kinds should preserve the `LO`, version, kind prefix so firmware can
quickly reject unsupported traffic.
